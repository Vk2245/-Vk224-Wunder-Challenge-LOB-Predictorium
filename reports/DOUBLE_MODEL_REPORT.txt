SECTION 1: THE PROBLEM - Why We Initially Failed
================================================================================

Initial Approach (Days 1-3):
----------------------------
- Trained 17 tree-based models (XGBoost, LightGBM, CatBoost)
- Engineered 128 hand-crafted features
- Achieved 0.2562 validation score (offline)
- Competition submissions: 0.006 average (essentially zero)

Root Cause Analysis:
-------------------
The competition uses STREAMING MODE inference:
- Server calls predict() for each row individually
- Only last 100 rows available in rolling buffer
- Cannot use groupby().transform() operations
- Features computed on 100 rows ≠ features on 1000 rows

Example of the Mismatch:
Training:   mean_price = df.groupby('seq_ix')['price'].mean()  # All 1000 rows
            → Result: 100.5 (consistent across sequence)

Inference:  mean_price = buffer[-100:]['price'].mean()  # Only 100 rows
            → Result: 98.2 at step 100, 102.1 at step 500 (varies!)

Models trained on full-sequence statistics received completely different
inputs during streaming inference → predictions became meaningless.

Additional Issues:
- Feature engineering took 50ms per call
- 901 predictions/sequence × 50ms = 45 seconds/sequence
- 10,721 sequences × 45s = 134 hours (way over 60-min limit)

Result: First 4 submissions either timed out or scored near zero.

SECTION 2: THE SOLUTION - Sequence Models for Streaming
================================================================================

Why Sequence Models Work:
-------------------------
Sequence models (RNN, GRU, LSTM, Transformer, MLP-based) are designed for
streaming data:

1. Process data SEQUENTIALLY (one step at a time)
2. Maintain INTERNAL STATE (memory of past steps)
3. Work on RAW FEATURES (no hand-engineered statistics)
4. Training = Inference (same rolling window process)

Key Insight: NO FEATURE ENGINEERING
-----------------------------------
Instead of computing statistics, we feed raw LOB data directly:
- Input: Last 100 rows of [p0..p11, v0..v11, dp0..dp3, dv0..dv3] (32 features)
- Shape: (100, 32) tensor
- Only normalization: (x - mean) / std
- Model learns what patterns matter

Training Process = Inference Process:
-------------------------------------
Training:
    X = raw_data[i:i+100]  # Any 100-step window
    X_norm = (X - mu) / sig
    prediction = model(X_norm)

Inference (streaming):
    buffer.append(new_row)
    X = np.stack(buffer[-100:])  # Last 100 rows
    X_norm = (X - mu) / sig
    prediction = model(X_norm)  # EXACT SAME PROCESS!

No mismatch! Features are identical!

SECTION 5: OPTIMIZATION - INT8 Quantization
================================================================================

Why Quantization Was Critical:
------------------------------
Initial 2-model ensemble (BiGRU + TimeMixer FP32):
- Speed: 242 pred/s
- Estimated runtime: 89 minutes
- Result: TIMEOUT on competition server

Competition Requirements:
- Must finish in < 60 minutes
- Running on 1 vCPU (very limited compute)
- 1.3 million predictions required

INT8 Quantization Process:
--------------------------
Tool: ONNX Runtime quantize_dynamic()
Method: Dynamic quantization (weights only)
Command: python quantize_models.py

Results:
- Original (FP32): 1.14 MB
- Quantized (INT8): 0.34 MB
- Size reduction: 70.7%
- Speed improvement: 3-4x faster
- Accuracy loss: < 1%

How INT8 Works:
--------------
FP32 (32-bit floating point):
- Range: ±3.4 × 10^38
- Precision: 7 decimal digits
- Memory: 4 bytes per weight

INT8 (8-bit integer):
- Range: -128 to 127
- Precision: Integer only
- Memory: 1 byte per weight

Quantization Formula:
    quantized_value = round((fp32_value - zero_point) * scale)
    
During inference:
    fp32_value = (quantized_value / scale) + zero_point

Benefits:
- 4x less memory (1 byte vs 4 bytes)
- Faster CPU operations (integer math vs floating point)
- Better cache utilization (more weights fit in cache)
- Minimal accuracy loss (< 1% for neural networks)

Final Performance:
-----------------
Single Model (TimeMixer INT8):
- Speed: 1045 pred/s (4.3x faster than FP32 ensemble!)
- Estimated runtime: 21 minutes
- Competition runtime: ~15-20 minutes (actual)
- Score: 0.2685

SECTION 6: IMPLEMENTATION DETAILS
================================================================================

Code Structure:
--------------
src/solution.py - Main inference code
models/timemixer_int8.onnx - Quantized model (0.34 MB)
models/scaler_tm.npz - Normalization parameters (mu, sig)

Key Optimizations in solution.py:
---------------------------------

1. Pre-computed Inverse for Division:
   self.inv_sig = (1.0 / sc["sig"]).astype(np.float32)
   # Multiplication is faster than division
   x_norm = (x - mu) * inv_sig  # Instead of (x - mu) / sig

2. Pre-allocated Arrays:
   self.x_buf = np.empty((SEQ_LEN, 32), dtype=np.float32)
   self.x_in = np.empty((1, SEQ_LEN, 32), dtype=np.float32)
   # Avoid repeated memory allocation

3. In-place Operations:
   np.subtract(self.x_buf, self.mu, out=self.x_in[0])
   np.multiply(self.x_in[0], self.inv_sig, out=self.x_in[0])
   # Reuse memory, avoid creating new arrays

4. ONNX Runtime Optimizations:
   so.intra_op_num_threads = 1  # Single thread for 1 vCPU
   so.inter_op_num_threads = 1
   so.graph_optimization_level = ORT_ENABLE_ALL
   so.execution_mode = ORT_SEQUENTIAL

5. Efficient Buffer Management:
   self.buf = deque(maxlen=SEQ_LEN)  # Automatic FIFO
   for i, x in enumerate(self.buf):
       self.x_buf[i] = x  # Direct copy, no np.stack overhead

Inference Flow:
--------------
1. Receive DataPoint (seq_ix, step, need_prediction, state)
2. Check if new sequence → reset buffer if needed
3. Append current state to rolling buffer
4. If not prediction step → return None
5. If buffer < 100 rows → return zeros
6. Stack buffer into pre-allocated array
7. Normalize: (x - mu) * inv_sig
8. Run ONNX inference
9. Return prediction as float64

Performance Characteristics:
---------------------------
- Memory usage: ~50 MB (model + buffers)
- CPU usage: 100% of 1 core (single-threaded)
- Inference time: ~0.96 ms per prediction
- Throughput: 1045 predictions/second
- Total time: 1.3M predictions / 1045 pred/s = 1244 seconds = 21 minutes

-----------------------------------
Problem: Test script took too long
Solution: Pre-convert to numpy arrays, iterate over arrays
Result: 10-20x faster testing

Challenge 5: Memory Allocation Overhead
---------------------------------------
Problem: Repeated np.stack() and array creation
Solution: Pre-allocate arrays, use in-place operations
Result: 10-15% speedup

----------------------------------

Solution: Removed torch.compile(), used standard PyTorch
Result: Training worked on Windows

-----------------------------------------
